# コンテンツ更新・メンテナンス設計書

## 1. 目的

本書は、WordPressへ投稿した記事やアプリ内で完了済みの記事を、公開後も品質、正確性、SEO価値、安全性を維持するための更新・再検証・統合・削除方針を定義する。

対象は、更新対象判定、リスク区分別の更新頻度、古い情報の再取得、X引用の再検証、Search Console等の指標確認、WordPress再投稿、更新日変更、統合・削除・noindex判断、監査ログ、テスト観点である。

## 2. 基本方針

- 公開後の記事品質は、作成時の品質基準だけでなく、更新性、正確性、実質改善、読者価値で判断する。
- 更新日だけを変更し、本文や根拠を実質更新しない運用を禁止する。
- `strict`、`compliance_strict`、X引用ありの記事は、公開後も再取得と人間確認を重視する。
- 価格、仕様、在庫、ランキング、口コミ、API料金、法律、医療、投資、政治、安全、評判は古くなりやすい情報として扱う。
- Search Console、アクセス解析、CV、指名検索、AI露出は更新対象を決める補助指標として扱う。
- MVPではSearch Console、GA4、WordPress Analytics APIの自動連携は対象外とし、手動確認またはCSV等の外部入力を後続フェーズで検討する。
- 本文履歴はMVPでは保持しない。更新監査は操作結果、理由、件数、状態変更だけを記録する。

## 3. 対象記事

メンテナンス対象は以下とする。

| 対象 | 扱い |
| --- | --- |
| `Completed`記事 | 投稿前の再評価対象 |
| `Posted`記事 | 公開後メンテナンス対象 |
| WordPress下書き投稿済み記事 | 投稿先との差分確認対象 |
| WordPress公開済み記事 | 再投稿、更新、統合、削除判断対象 |
| `HumanReviewRequired`記事 | 人間確認が終わるまでPublish不可 |
| 論理削除済み記事 | 通常メンテナンス対象外 |

アプリ内の記事が削除済みでも、WordPress側に公開記事が残る可能性がある。WordPress側の削除やnoindexはMVPでは自動実行せず、操作手順と確認項目を提示する。

## 4. メンテナンス状態

記事のメンテナンス状態は以下を想定する。

| 状態 | 意味 | 次の操作 |
| --- | --- | --- |
| `Fresh` | 現時点で更新不要 | 通常監視を続ける |
| `ReviewDue` | 定期確認時期に到達 | 内容、出典、指標を確認する |
| `NeedsRefresh` | 情報再取得が必要 | Tavily、X、公式情報を再取得する |
| `NeedsRevision` | 本文修正が必要 | リライト、追記、削除、統合を行う |
| `HumanReviewRequired` | 自動判断だけで公開更新不可 | 人間確認を行う |
| `ArchiveCandidate` | 統合、削除、noindex候補 | SEO影響と内部リンクを確認する |

MVPでは専用Entityを追加しない。記事状態、`TopicRisk`、`HumanReviewRequired`、ジョブ結果、監査ログ、画面上の警告で表現する。

## 5. 更新対象判定

以下に該当する記事は更新対象にする。

| 条件 | 判定 |
| --- | --- |
| `strict`記事の最終確認から所定期間が経過 | `ReviewDue` |
| `compliance_strict`記事の最終確認から所定期間が経過 | `HumanReviewRequired` |
| 価格、仕様、在庫、ランキング、API料金を含む | `NeedsRefresh` |
| X引用、口コミ、評判、SNS反応を含む | `NeedsRefresh` |
| 出典URLが404、リダイレクト、内容変更 | `NeedsRevision` |
| Search Consoleで表示回数はあるがCTRが低い | タイトル、メタディスクリプション見直し |
| 流入やCVが大きく低下 | 内容、検索意図、競合差分を確認 |
| 重複または薄い記事が複数ある | `ArchiveCandidate` |
| 法律、医療、金融、政治、安全に関する変更が発生 | `HumanReviewRequired` |
| WordPress投稿後に表示崩れや危険HTMLが見つかる | `NeedsRevision` |

検索順位だけで更新対象を決めない。読者価値、CV、検索意図、情報鮮度、重複、事業上の重要度を合わせて判断する。

## 6. リスク区分別の更新頻度

更新頻度は[トピックリスク分類設計書](topic-risk-classification.md)の`TopicRisk`を基準にする。

| TopicRisk | 定期確認 | 再取得 | 公開更新 |
| --- | --- | --- | --- |
| `normal` | 3から6か月ごと | 必要時 | 品質改善時に更新 |
| `strict` | 1か月ごと、または変化検知時 | 価格、仕様、X引用、口コミは必須 | 取得日と変更点を確認後に更新 |
| `compliance_strict` | 1か月ごと、または関連変更検知時 | 公式・一次情報を優先 | 人間確認後のみ公開更新 |

緊急性が高い変更がある場合は、定期確認日を待たずに更新する。

## 7. 情報種別ごとの再検証

| 情報種別 | 再検証内容 | 更新判断 |
| --- | --- | --- |
| 価格、料金、API料金 | 公式料金、取得日、通貨、税、プラン | 差分があれば更新 |
| 仕様、機能、制限 | 公式ドキュメント、リリースノート | 廃止、非推奨、GA化があれば更新 |
| 在庫、販売状況 | 公式商品ページ、販売ページ | 在庫・販売終了なら更新 |
| ランキング | 評価基準、対象期間、集計方法 | 根拠が古い場合は更新または削除 |
| 口コミ、評判 | X投稿、レビュー、公式発表 | 個別投稿の断定を避けて更新 |
| 法律、税金、投資、医療 | 公的機関、公式、専門家確認 | 人間確認なしに断定しない |
| 政治、事件、不祥事 | 一次情報、複数出典、更新日 | 名誉毀損リスクを確認 |
| SEO情報 | Google公式、Search Central、更新日 | 古いハックや無効施策を削除 |

外部情報を再取得できない場合は、最新情報として断定しない。必要に応じて「確認時点」「公式情報を確認する」などの前提を本文へ追加する。

## 8. SEOメンテナンス

SEOメンテナンスでは以下を確認する。

| 項目 | 確認内容 |
| --- | --- |
| 検索意図 | 現在の検索意図と記事内容が一致するか |
| タイトル | 内容、検索意図、独自価値を正確に示すか |
| メタディスクリプション | 本文と矛盾せず、クリック理由があるか |
| 見出し | 結論、理由、条件、比較、次の行動が追えるか |
| 独自情報 | 一次体験、実測値、比較条件、判断基準があるか |
| E-E-A-T | 著者、根拠、更新日、作成方法が確認できるか |
| 内部リンク | 関連記事へ文脈に沿ってリンクしているか |
| 重複 | 同じ検索意図の記事が乱立していないか |
| 技術SEO | noindex、canonical、robots、表示崩れがないか |

AI検索時代でも基本SEOを前提とする。AI向けの不自然な細切れ文章、検索語バリエーション別の薄いページ量産、特殊マークアップ依存は更新方針にしない。

## 9. Search Consoleとアクセス解析

MVPではSearch ConsoleやGA4のAPI連携を実装しない。運用では手動確認またはエクスポート値を参照する。

確認する指標:

| 指標 | 使い方 |
| --- | --- |
| 表示回数 | 検索面で露出があるかを見る |
| CTR | タイトル、説明、検索意図のずれを見る |
| 平均掲載順位 | 大きな下落や競合変化を確認する |
| クエリ | 想定外の検索意図を検出する |
| ランディングページ | 更新優先度を決める |
| CV | 事業成果に結びつく記事を優先する |
| 滞在、再訪 | 読者満足の補助指標にする |
| 指名検索 | ブランド認知の変化を見る |

Search ConsoleのAI機能由来データは通常のWeb検索タイプへ含まれる前提で扱う。AI露出を個別に追う場合は、手動確認、外部計測、アクセス解析、CVを組み合わせる。

## 10. 更新アクション

更新対象記事には以下のアクションを選ぶ。

| アクション | 適用条件 | 注意 |
| --- | --- | --- |
| `Refresh` | 古い情報、価格、仕様、口コミの再取得 | 再取得できない情報を最新と断定しない |
| `Rewrite` | 読みにくい、重複、検索意図ずれ | 意味と根拠を保つ |
| `Expand` | 独自情報や判断基準が不足 | 架空の体験談を追加しない |
| `Summarize` | 冗長、重複、薄い追記が多い | 注意事項を落とさない |
| `Merge` | 同じ検索意図の記事が複数ある | 統合先、内部リンク、リダイレクトを確認 |
| `Archive` | 古く、価値が低く、更新困難 | 削除、noindex、リダイレクトを検討 |
| `NoAction` | 変化なし、品質維持 | 確認日と理由だけ残す |

記事を増やすだけでなく、統合、削除、noindexを選べるようにする。

## 11. 更新日を変更できる条件

更新日は、読者にとって意味のある変更がある場合だけ変更する。

更新日変更を許可する例:

- 価格、仕様、在庫、ランキング、法律、API料金を再確認し、本文へ反映した。
- 古い出典を新しい公式情報へ差し替えた。
- X引用を再取得し、削除・非公開・編集の有無を確認した。
- 独自情報、実測値、比較条件、判断基準を追加した。
- 検索意図に合わせてタイトル、見出し、本文を実質的に改善した。
- 誤情報、古い断定、危険表現を修正した。

更新日変更を禁止する例:

- 誤字だけを直した。
- 内部メモだけを更新した。
- 日付だけを新しくした。
- 自動生成で薄いFAQを追加しただけである。
- 出典や本文を確認せずに「最新版」と書いた。

軽微な修正は監査ログに記録できるが、公開記事の更新日を変える根拠にしない。

## 12. WordPress再投稿・差し替え

WordPress投稿済み記事を更新する場合は以下を確認する。

1. アプリ内記事本文を更新する。
2. MarkdownからHTMLへ再変換する。
3. HTMLを再サニタイズする。
4. `TopicRisk`と品質チェックを再実行する。
5. X引用がある場合は再取得する。
6. WordPress投稿先の所有者を確認する。
7. 投稿IDまたは投稿URLを確認する。
8. 下書き更新または公開済み記事の差し替えを行う。
9. 投稿結果、更新理由、更新時刻を監査ログへ記録する。

MVPではWordPress側の既存公開記事を自動で削除しない。削除、リダイレクト、noindex、canonical変更は手動確認または後続フェーズの対象とする。

## 13. 統合・削除・noindex判断

以下に該当する記事は統合、削除、noindex候補とする。

| 条件 | 推奨判断 |
| --- | --- |
| 同じ検索意図の記事が複数ある | 統合 |
| 古く、更新しても読者価値が低い | 削除またはnoindex |
| 外部情報の再取得ができず正確性を保てない | 公開停止または注意書き |
| ほぼ検索流入がなく、内部リンクも少ない | 統合または削除候補 |
| 古いランキングや比較が誤解を生む | 更新、統合、またはnoindex |
| 法的・医療的リスクが高く監修不能 | 公開停止候補 |

削除前には、内部リンク、外部リンク、CV、検索流入、WordPress側URL、リダイレクト要否を確認する。

## 14. X引用の再検証

X引用がある記事では以下を確認する。

| 確認項目 | 方針 |
| --- | --- |
| 投稿の存在 | 再取得できない場合は引用停止 |
| 公開状態 | 非公開、削除、凍結の場合は引用停止 |
| 編集有無 | 内容が変わった場合は本文文脈を再確認 |
| 投稿日時 | 本文の時系列と矛盾しないか確認 |
| 投稿URL | `https://x.com/...`または`https://twitter.com/...`のみ許可 |
| 個人攻撃 | 誹謗中傷や差別を増幅しない |

X投稿は引用元として表示できるが、事実確定の単独根拠にしない。X引用の再取得に失敗した場合は、引用停止または`HumanReviewRequired`へ上げる。

## 15. compliance_strict記事の更新

`compliance_strict`記事では以下を必須とする。

- 人間確認前にPublishしない。
- 公式、一次情報、公的機関、専門家確認を優先する。
- 医療、法律、税金、投資、保険、政治、安全、事件、評判に関する断定を避ける。
- 読者へ直接的な医療行為、投資行動、法的判断、投票行動を促さない。
- 更新後も著者、監修者、出典、更新日、注意事項を確認する。
- 企業や個人の評判に関する記述は名誉毀損リスクを確認する。

人間確認が完了していない場合、WordPress公開済み記事の更新は下書き差し替えまたは公開停止検討に留める。

## 16. 通知

以下のイベントでは通知対象にできる。

| イベント | 通知先 | 内容 |
| --- | --- | --- |
| 更新期限到達 | 管理者または記事所有者 | 記事ID、タイトル、理由 |
| X再取得失敗 | 管理者または記事所有者 | 投稿URL、エラー種別、対応要否 |
| `HumanReviewRequired`発生 | 管理者または記事所有者 | 対象記事、理由 |
| WordPress更新成功 | 記事所有者 | 投稿URL、更新時刻 |
| WordPress更新失敗 | 記事所有者 | ErrorCode、traceId |

通知には本文全文、プロンプト全文、X投稿本文全文、秘密情報を含めない。

## 17. 監査ログ

監査ログに記録する操作:

- 更新対象判定
- 品質チェック実行
- X再取得結果
- 人間確認完了
- WordPress下書き更新
- WordPress公開更新
- 統合、削除、noindex判断
- 更新日変更
- 手動でのメンテナンス状態変更

記録する情報:

| 項目 | 内容 |
| --- | --- |
| `articleId` | 対象記事ID |
| `wordpressPostId` | ある場合のみ |
| `eventName` | 操作名 |
| `reasonCode` | 更新理由 |
| `topicRisk` | 判定区分 |
| `humanReviewRequired` | 人間確認要否 |
| `result` | 成功、失敗、保留 |
| `traceId` | 調査用ID |

本文全文、HTML本文全文、プロンプト全文、外部APIレスポンス全文は監査ログへ保存しない。

## 18. エラー処理

メンテナンス専用ErrorCodeはMVPでは追加しない。既存コードを使う。

| 状況 | ErrorCode | 方針 |
| --- | --- | --- |
| 記事が存在しない | `NotFound` | 更新処理を拒否する |
| 権限がない | `Forbidden` | 更新処理を拒否する |
| 投稿可能状態ではない | `ArticleNotPostable` | WordPress更新を拒否する |
| 人間確認が必要 | `HumanReviewRequired` | Publishを拒否する |
| X再取得が必要 | `XRehydrationRequired` | 再取得を要求する |
| X再取得失敗 | `XRehydrationFailed` | 引用停止または人間確認へ回す |
| HTMLサニタイズ後に空 | `ValidationError` | 更新を拒否する |
| WordPress更新失敗 | 外部連携系ErrorCode | 再試行または手動対応 |

`ErrorMessage`には利用者に表示できる短い理由だけを保存する。

## 19. ログ

ログへ出してよい情報:

- `traceId`
- `userId`
- `articleId`
- `jobId`
- `eventName`
- `topicRisk`
- `maintenanceStatus`
- `reasonCode`
- 文字数、見出し数、リンク数、引用数
- 更新前後の状態名

ログへ出さない情報:

- 記事本文全文
- HTML本文全文
- プロンプト全文
- X投稿本文全文
- 外部APIレスポンス全文
- WordPress Application Password
- Discord Webhook URL

## 20. テスト観点

| ID | 観点 | 期待結果 |
| --- | --- | --- |
| `CUM-001` | normal定期確認 | 期限到達で`ReviewDue`になる |
| `CUM-002` | strict期限到達 | 価格や仕様記事が`NeedsRefresh`になる |
| `CUM-003` | compliance_strict更新 | 人間確認前のPublish更新が拒否される |
| `CUM-004` | 更新日変更許可 | 実質更新がある場合だけ更新日変更が許可される |
| `CUM-005` | 更新日変更拒否 | 日付だけ更新が拒否または警告される |
| `CUM-006` | X再取得成功 | X引用が再取得後に維持される |
| `CUM-007` | X再取得失敗 | 引用停止または`HumanReviewRequired`になる |
| `CUM-008` | WordPress再投稿 | サニタイズ済みHTMLで既存投稿が更新される |
| `CUM-009` | 統合候補 | 同一検索意図の記事が`ArchiveCandidate`になる |
| `CUM-010` | ログ除外 | 本文全文、プロンプト全文、X投稿本文全文がログに出ない |
| `CUM-011` | Search Console手動入力 | 低CTR記事がタイトル見直し候補になる |
| `CUM-012` | noindex候補 | 古く価値が低い記事が公開停止候補になる |

テストではGemini、Tavily、X API、WordPress、Search Console、GA4の実APIを呼ばない。

## 21. 実装順序

1. メンテナンス状態の判定型をApplication層に定義する。
2. `TopicRisk`別の確認期限計算を実装する。
3. 更新対象判定サービスを実装する。
4. X引用再取得チェックをメンテナンス処理へ組み込む。
5. 品質チェックとHTML再サニタイズを更新前に実行する。
6. WordPress既存投稿更新処理を実装する。
7. 更新日変更可否の判定を実装する。
8. 監査ログと通知を実装する。
9. `CUM-*`テストを追加する。

Search Console、GA4、WordPress側のnoindex/canonical/redirect操作の自動化は後続フェーズとする。

## 22. 決定事項

- 更新日だけを変更する運用は禁止する。
- MVPでは本文履歴を保持しない。
- MVPではSearch Console、GA4、WordPress Analytics APIの自動連携を実装しない。
- `strict`記事は最新性、出典、X引用、価格、仕様を重点確認する。
- `compliance_strict`記事は人間確認前にPublish更新しない。
- X引用は公開後更新時にも再取得し、取得不能なら引用停止または人間確認へ回す。
- WordPress側の削除、noindex、canonical、redirectの自動操作は後続フェーズとする。
