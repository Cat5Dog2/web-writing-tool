# 記事品質ガイドライン

## 1. 目的

本書は、AI生成記事、リライト記事、再取得後の記事を公開可能な品質へ揃えるための基準を定義する。

対象は、記事タイトル、メタディスクリプション、見出し構成、本文、引用元、X投稿引用、SEO、事実確認、公開前レビュー、差し戻し判定である。

## 2. 基本方針

- 記事品質は、読みやすさ、検索意図への適合、事実性、安全性、公開後の信頼性で判断する。
- AI生成結果は下書きとして扱い、公開可能性を自動判定または人間確認で判断する。
- 検索結果、X投稿、ユーザー入力、WordPress由来データを事実根拠として無条件に信頼しない。
- 出典が不足する内容、最新性が必要な内容、専門的判断を含む内容は断定しない。
- 著作権、名誉毀損、医療、法律、税金、投資、政治、安全、評判に関わる内容は慎重に扱う。
- `compliance_strict`または`HumanReviewRequired`の記事は人間確認前に公開投稿しない。
- WordPress投稿の既定ステータスは`Draft`とし、公開投稿は明示操作と確認を必要とする。
- SEOは検索順位操作ではなく、読者に役立つ独自情報を検索エンジンとAI検索が理解しやすい形で提供する活動として扱う。
- 2026年5月時点では、AI OverviewsやAI Mode向けにも基本SEO、技術的なクロール可能性、非コモディティな内容、E-E-A-Tを重視する。
- `llms.txt`、AI専用マークアップ、AI向けの不自然な細切れ文章、検索語バリエーション別の薄いページ量産を品質向上策として扱わない。

## 3. 品質状態

記事品質の判定状態は以下を基本とする。

| 状態 | 意味 | 次の操作 |
| --- | --- | --- |
| `Pass` | 公開前チェックを満たす | 下書き投稿または公開操作へ進める |
| `NeedsRevision` | 修正すれば公開可能 | リライト、再取得、手動編集を行う |
| `HumanReviewRequired` | 自動判定だけで公開不可 | 人間確認を必須にする |
| `Reject` | 公開に不適切 | 記事の再作成または削除を検討する |

MVPでは専用Entityを追加しない。記事状態、`TopicRisk`、`HumanReviewRequired`、ジョブ結果、画面上の警告で表現する。

## 4. 品質観点

| 観点 | 合格基準 | 不合格例 |
| --- | --- | --- |
| 検索意図 | キーワードに対する読者の疑問へ直接答える | 関係ない一般論が多い |
| タイトル | 内容を正確に表し、過剰な煽りがない | 根拠のない最上級、クリックベイト |
| 見出し | H2/H3が論理的で重複しない | 同じ論点が複数見出しに分散する |
| 本文 | 見出しに対応し、具体性と読みやすさがある | 薄い言い換え、冗長、主題逸脱 |
| 事実性 | 根拠が必要な情報に出典または確認条件がある | 数値、価格、日付を根拠なく断定する |
| 安全性 | 危険表現、秘密情報、危険HTMLを含まない | APIキー、個人情報、`script`を含む |
| 独自性 | 外部本文の丸写しではなく要約・解釈になっている | 検索結果や外部記事の長文コピー |
| 独自価値 | 一次体験、実測値、比較条件、判断基準がある | 誰でも書ける一般論だけで終わる |
| E-E-A-T | 著者、根拠、作成方法、専門性、更新日が確認できる | 誰が、どう、なぜ書いたか不明 |
| AI検索適合 | 結論、根拠、条件、比較が明確で引用元として使いやすい | AI向けだけを狙った不自然な分割や量産 |
| WordPress適合 | サニタイズ済みHTMLへ変換できる | 許可外HTMLや空本文になる |

## 5. タイトル品質

タイトルは以下を満たす。

- 記事本文の内容と一致する。
- 主要キーワードを自然に含める。
- 検索意図と独自価値を同時に示す。
- 読者が得られる情報を具体的に示す。
- 根拠のない「完全」「絶対」「必ず」「最強」「最新」を避ける。
- すべての記事で似た定型文を使い回さない。
- 価格、ランキング、口コミ、評判など変動しやすい内容は取得日や前提を本文側で補足できる場合のみ扱う。
- 医療、法律、税金、投資、保険、政治、安全、事件、評判では断定的な誘導を避ける。

不適切な例:

- `絶対に失敗しない投資方法`
- `最安のサービスを完全比較`
- `この病気は自宅で治せる`

望ましい例:

- `WordPress下書き投稿を安全に自動化する設計ポイント`
- `料金比較記事で確認すべき更新日と注意点`
- `X投稿を記事の引用元として扱うときの確認観点`
- `SEOライティングの正解は検索意図と独自情報を両立すること`
- `AI検索時代の記事品質チェックリスト`

## 6. メタディスクリプション品質

メタディスクリプションは以下を満たす。

- 記事の要点を短く説明する。
- 本文にない内容を追加しない。
- 誇張や煽りを避ける。
- 読者にとっての読む理由を明確にする。
- 誰が読むと何を得られるかを示す。
- 必要に応じて著者、公開日、更新日、対象範囲などの判断材料を含める。
- 検索結果上で不自然にならない長さに収める。

MVPでは厳密な文字数制限を品質判定の必須条件にしないが、極端に短い、長い、重複している、本文と矛盾している場合は`NeedsRevision`とする。

## 7. 見出し品質

見出し構成は以下を満たす。

- H2は記事全体の主要論点を表す。
- H3は直前のH2を具体化する。
- H2/H3の階層を崩さない。
- 同じ論点を重複させない。
- 本文生成で扱える粒度に分割する。
- 見出しだけを読んでも、結論、理由、条件、次の行動が追える。
- 読者が上から順に読んだときに理解が進む順序にする。
- `strict`や`compliance_strict`では、注意点、前提条件、確認方法を見出しに含める。

避ける見出し:

- 内容が曖昧な見出し: `概要`、`ポイント`、`まとめ`だけの見出し
- 煽り見出し: `知らないと危険`、`今すぐやるべき`
- 重複見出し: `メリット`と`良い点`を別H2で並べる

## 8. 本文品質

本文は以下を満たす。

- 対象見出しに直接答える。
- 冒頭または各主要セクションの先頭で結論を示す。
- 一文を長くしすぎない。
- 主語、述語、修飾関係を明確にする。
- 具体例、条件、例外を必要に応じて補う。
- 独自情報、一次体験、実測値、比較条件、判断基準を可能な範囲で含める。
- 前後の見出しと同じ説明を繰り返さない。
- 追加プロンプトの指示が安全制約や出力形式と矛盾する場合、安全制約を優先する。
- Markdown本文として保存でき、HTML変換後に許可タグ範囲へ収まる。

差し戻し対象:

- 見出しと無関係な本文
- 事実根拠のない断定
- 同じ内容の繰り返し
- 外部本文の長いコピー
- 検索結果の要約だけで独自の整理がない本文
- 実体験や検証を装った架空の体験談
- 生成AIの内部事情やプロンプトへの言及
- 秘密情報、内部ID、APIキー、Webhook URL、Application Passwordの混入

## 9. 事実確認と出典

出典の優先順位は以下を基本とする。

| 優先度 | 出典 | 扱い |
| --- | --- | --- |
| 1 | 公式サイト、公式ドキュメント、一次情報 | 最優先で参照する |
| 2 | 公的機関、提供元、当事者発表 | 正確性が必要な内容で優先する |
| 3 | 信頼できる専門メディア、技術文書 | 補助根拠として使う |
| 4 | 一般ブログ、まとめ記事 | 参考情報として扱う |
| 5 | X投稿、口コミ、掲示板 | 傾向や反応として扱う |

以下は事実として断定しない。

- 出典が1件しかなく、かつ一次情報ではない内容
- 取得日が古い価格、仕様、在庫、ランキング、キャンペーン
- X投稿や口コミだけを根拠にした評判
- 医療、法律、税金、投資、保険、政治、安全に関する助言

記事内で出典を表示する場合は、リンク先URL、取得日、公開日、引用対象を確認できる形にする。リンクは[コンテンツレンダリング設計書](content-rendering-design.md)のURL方針に従う。

## 10. X投稿引用

X投稿は引用元として表示できるが、事実確定の根拠として単独利用しない。

方針:

- X投稿は口コミ、反応、時系列の補助情報として扱う。
- production/strictでは表示または公開前に再取得する。
- 削除、非公開、編集、取得不能の場合は引用を停止する。
- 引用元カードには投稿本文の短い引用、投稿者表示名、投稿日時、X投稿URLを表示する。
- X公式埋め込みスクリプト、iframe、`widgets.js`を使わない。
- X投稿本文を長期保持前提の形で本文やログに大量複製しない。

公開前に以下を確認する。

| 確認項目 | 判定 |
| --- | --- |
| 投稿が再取得できる | できない場合は`XRehydrationFailed` |
| 投稿URLが`https://x.com/...`または`https://twitter.com/...` | それ以外は拒否 |
| 投稿内容が記事文脈と一致する | 一致しない場合は削除または差し替え |
| 個人攻撃、差別、誹謗中傷を増幅しない | 問題がある場合は引用しない |

## 11. SEO品質

SEO品質は検索エンジン向けの技巧ではなく、読者が必要な情報へ到達しやすい構造として扱う。

2026年5月時点の方針:

- AI検索でも基本SEOを前提にする。
- 検索者の疑問へ最短で答え、根拠、条件、比較、次の行動を示す。
- 共通情報の焼き直しではなく、一次情報、経験、検証、独自判断を含める。
- E-E-A-Tは本文、著者情報、出典、更新日、作成方法で読者に見える形にする。
- AI生成は下書き、構成案、校正支援として使い、公開前に人間または品質チェックで編集・検証する。
- 検索順位やAI応答を操作する目的の大量生成、薄い派生ページ、キーワード詰め込みを行わない。
- AI OverviewsやAI Modeだけを狙った特殊マークアップ、`llms.txt`依存、AI向けの不自然な`chunking`を品質基準にしない。

### 11.1 検索意図

検索意図はキーワードそのものではなく、読者が解決したい課題として扱う。

| 意図 | 例 | 記事で必要な答え |
| --- | --- | --- |
| `Know` | SEOライティングとは | 定義、仕組み、背景、注意点 |
| `Do` | SEO記事 書き方 | 手順、テンプレート、チェックリスト |
| `Buy` | SEOライティング 外注 比較 | 比較条件、料金確認、選び方、リスク |
| `Go` | Google Search Console ログイン | 公式導線、操作手順、確認方法 |

同じ記事内で複数の検索意図を扱う場合は、主意図を明確にし、補助意図は見出しやFAQで補う。主意図が曖昧な記事は`NeedsRevision`とする。

### 11.2 SEOライティングの最低基準

- タイトル、メタディスクリプション、H2/H3、本文が同じ主題を扱う。
- 冒頭で記事の対象、前提、結論がわかる。
- 見出しだけで記事の流れと答えを把握できる。
- 読者の次の行動に必要な情報が本文に含まれる。
- 不自然なキーワード詰め込みをしない。
- 本文にない内容をタイトルやメタディスクリプションで約束しない。
- 重要な事実、数値、比較条件に出典または確認方法がある。
- 更新性が必要な情報には取得日、更新日、前提条件を示す。
- 内部リンクは「詳しくはこちら」ではなく、リンク先の内容がわかるアンカーテキストにする。

### 11.3 独自情報と非コモディティ

以下は避ける。

- 同じキーワードの過剰反復
- 根拠のないランキング形式
- 競合記事の構成だけをなぞった薄い本文
- 検索結果の要約だけで独自の整理がない本文
- AIで大量生成しただけの本文
- 更新日だけを変え、内容を実質更新しない本文
- FAQや関連質問を検索流入目的だけで量産した本文

入れるべき独自情報:

| 種別 | 内容 |
| --- | --- |
| 一次体験 | 実際に使った結果、導入前後、失敗談、運用上の注意 |
| 実測値 | 速度、費用、期間、順位推移、CVR、比較条件 |
| 自社データ | アンケート、問い合わせ傾向、売上変化、利用ログからの集計 |
| 専門家視点 | 業界経験、資格、監修、判断基準 |
| 具体例 | 実際の手順、設定例、スクリーンショット、表、チェックリスト |
| 意思決定 | 向いている人、向かない人、選ばない条件 |

MVPでは画像を本文HTMLへ含めないため、画像・図解・スクリーンショットは後続フェーズの品質強化項目とする。本文では図解候補や表現方針をメモとして扱える。

### 11.4 E-E-A-TとWho / How / Why

E-E-A-Tは機械的なスコアではなく、読者が信頼できるかを判断する材料として扱う。

| 観点 | 記事内またはサイト内で示す内容 |
| --- | --- |
| Experience | 実体験、使用期間、検証条件、現場事例 |
| Expertise | 著者の専門領域、資格、経験、監修者 |
| Authoritativeness | 実績、掲載、被引用、業界内での認知 |
| Trustworthiness | 出典、更新日、会社情報、問い合わせ先、リスク、広告表記 |
| Who | 誰が書いたか、誰が確認したか |
| How | どのように調査、生成、検証、編集したか |
| Why | 検索流入ではなく、読者の課題解決を目的としているか |

YMYL領域では信頼性を最優先し、著者、監修者、出典、更新日、注意事項のいずれかが欠ける場合は`HumanReviewRequired`とする。

### 11.5 AI検索への適合

AI OverviewsやAI Modeでは、記事全体だけでなく、回答に使いやすい結論、根拠、条件、比較、出典が重要になる。

適合基準:

- セクションごとに主張と根拠が対応している。
- 比較記事では比較条件、対象、評価基準、例外を明示する。
- 手順記事では前提、手順、確認方法、失敗時の戻し方を示す。
- レビュー記事では実体験、検証条件、利点、欠点、競合との差を示す。
- FAQは読者の追加疑問に答える場合だけ使う。
- 構造化データを使う場合は、画面上で見える本文と一致させる。

避ける方針:

- AI検索だけを狙った不自然な短文分割
- 似た検索語ごとの薄いページ量産
- AI専用schemaや`llms.txt`を主要なSEO施策として扱う
- 実在しない経験、出典、レビューの生成

### 11.6 技術SEOとWordPress投稿

記事品質チェックでは、本文品質だけでなくWordPress投稿後に検索エンジンが理解できる状態も確認する。

| 項目 | 品質基準 |
| --- | --- |
| クロール可能性 | 投稿先で`noindex`、認証、robots制限により意図せずブロックされていない |
| スニペット | title、meta description、本文冒頭が記事内容と一致する |
| 内部リンク | 関連記事へ文脈に沿ってリンクできる |
| canonical | 重複記事や再投稿がある場合に正規URLを確認する |
| 構造化データ | 表示本文と一致し、見えない情報だけを埋め込まない |
| ページ体験 | 広告、ポップアップ、表示崩れで本文閲覧を妨げない |

MVPではWordPressテーマ、canonical、構造化データ、Core Web Vitalsをアプリ側で完全制御しない。記事品質としては警告と確認項目に留める。

### 11.7 SEO評価指標

SEO評価は順位だけで判断しない。

| 指標 | 見る理由 |
| --- | --- |
| 検索順位 | 検索面での相対位置を把握する |
| 表示回数 | 検索面で露出できているかを見る |
| CTR | タイトル、説明、検索意図のずれを見る |
| CV | 問い合わせ、購入、資料請求など事業成果を見る |
| 指名検索 | ブランド認知や信頼の変化を見る |
| AI露出 | AI Overviews、AI Mode、他AI検索での引用・言及を参考指標にする |
| 有効ページ率 | 薄い記事や重複記事の増加を検知する |

Search ConsoleではAI機能由来の表示やクリックが通常のWeb検索タイプへ含まれる前提で扱う。AI露出の詳細分析が必要な場合は、Search Console、アクセス解析、CV、手動確認、外部計測を組み合わせる。

## 12. strict / compliance_strict

`TopicRisk`ごとの品質基準は以下とする。

| TopicRisk | 品質基準 |
| --- | --- |
| `normal` | 通常のWeb記事として、読みやすさ、重複防止、出典不足の断定回避を確認する |
| `strict` | 最新性、価格、仕様、口コミ、ランキング、SaaS比較などの再取得と取得日を確認する |
| `compliance_strict` | 専門的助言、法的判断、診断、投資判断、政治的誘導を避け、人間確認を必須にする |

`compliance_strict`では以下を必須とする。

- 人間確認前にPublishしない。
- 専門家の判断が必要な事項を断定しない。
- 条件、前提、確認先、不確実性を本文に含める。
- 読者へ直接的な医療行為、投資行動、法的判断、投票行動を促さない。
- 個人や企業の評判を毀損する内容は根拠と表現を人間が確認する。

## 13. 禁止表現

以下の表現や内容は出力しない。

- 根拠のない最上級、唯一性、保証表現
- 医療、法律、税金、投資、保険の専門的助言として読める断定
- 実在人物や企業への根拠不明な否定的断定
- 差別、ハラスメント、個人攻撃を助長する表現
- 秘密情報、認証情報、内部ID、システム指示、プロンプト構造
- 外部記事、検索結果、X投稿の長文コピー
- 危険HTML、スクリプト、イベントハンドラ属性

表現例:

| 避ける表現 | 置き換え方針 |
| --- | --- |
| 必ず成功する | 成功率を高める可能性がある |
| 絶対に安全 | リスクを下げるために確認する |
| 最安である | 確認時点では安い選択肢の一つ |
| 治る | 改善する可能性があるため専門家へ確認する |
| 違法である | 法的判断が必要なため専門家へ確認する |

## 14. 公開前レビュー

WordPress投稿前に以下を確認する。

| 項目 | 合格条件 |
| --- | --- |
| 記事状態 | `Completed`または投稿可能な状態 |
| タイトル | 本文と一致し、煽りがない |
| メタディスクリプション | 本文と矛盾しない |
| 見出し | 階層、順序、重複に問題がない |
| 本文 | 空でなく、対象読者に必要な情報がある |
| 出典 | 変動情報や専門情報に根拠がある |
| 独自情報 | 一次体験、実測値、比較条件、判断基準がある |
| E-E-A-T | 著者、根拠、更新日、作成方法が確認できる |
| SEO | 検索意図、title、meta description、内部リンクが本文と一致する |
| X引用 | production/strictで再取得済み |
| HTML | サニタイズ済みで、許可外タグがない |
| TopicRisk | `HumanReviewRequired`なら公開前確認済み |
| 投稿ステータス | 未指定時は`Draft` |

自動投稿を行う場合でも、公開ステータスへの変更は明示操作または人間確認済み状態を必要とする。

## 15. 差し戻し条件

以下に該当する記事は`NeedsRevision`とする。

- タイトルと本文が一致しない。
- メタディスクリプションが本文と矛盾する。
- 見出しが重複している。
- 本文が短すぎる、または冗長である。
- 出典が必要な主張に根拠がない。
- 独自情報がなく、検索結果や競合記事の要約だけで構成されている。
- E-E-A-Tを示す著者、根拠、更新日、作成方法のいずれかが不足している。
- title、meta description、見出し、本文の検索意図がずれている。
- X投稿の再取得が必要である。
- HTML変換後に許可外要素が除去され、本文の意味が崩れる。

以下に該当する記事は`HumanReviewRequired`とする。

- `TopicRisk = compliance_strict`
- 医療、法律、税金、投資、保険、政治、安全、事件、評判を扱う。
- 実在人物や企業への否定的評価を含む。
- X投稿や口コミを主要根拠として扱う。
- 最新性が重要だが再取得結果に不確実性がある。

以下に該当する記事は`Reject`とする。

- 秘密情報や認証情報を含む。
- 違法行為、危険行為、差別、ハラスメントを助長する。
- 外部本文の丸写しが大部分を占める。
- サニタイズ後に本文が空になる。
- 修正しても公開目的に合わない。

## 16. 品質チェックの実装方針

MVPでは以下の順で品質チェックを実装する。

1. 構造チェック
2. 文字数と空本文チェック
3. 危険HTMLチェック
4. 禁止表現チェック
5. SEO品質チェック
6. E-E-A-Tチェック
7. TopicRiskチェック
8. X引用再取得チェック
9. 人間確認必須判定

品質チェックはApplication層のサービスとして扱う。Web層は判定結果を表示し、Application層の判定を回避しない。

想定する判定結果:

| 項目 | 内容 |
| --- | --- |
| `status` | `Pass`、`NeedsRevision`、`HumanReviewRequired`、`Reject` |
| `reasons` | 利用者に表示できる短い理由 |
| `warnings` | 公開前に確認すべき警告 |
| `seoWarnings` | 検索意図、独自情報、E-E-A-T、内部リンク、更新性の警告 |
| `checkedAt` | UTC確認時刻 |

MVPでは品質結果を長期保存しない。後続フェーズで品質スコア履歴、差し戻し履歴、編集履歴を追加する。

## 17. エラー処理

品質チェック専用のErrorCodeはMVPでは追加しない。既存コードを使う。

| 状況 | ErrorCode | 方針 |
| --- | --- | --- |
| 必須項目不足 | `ValidationError` | チェックまたは投稿を拒否する |
| AI出力が構造不正 | `ExternalBadResponse` | 再試行または差し戻し |
| 投稿可能状態ではない | `ArticleNotPostable` | 投稿を拒否する |
| 人間確認が必要 | `HumanReviewRequired` | 公開投稿を拒否する |
| X再取得が必要 | `XRehydrationRequired` | 再取得を要求する |
| X再取得失敗 | `XRehydrationFailed` | 引用停止または人間確認へ回す |

`ErrorMessage`には利用者に表示できる短い理由だけを保存する。本文全文、プロンプト全文、外部APIレスポンス全文は保存しない。

## 18. ログ・監査

ログへ出してよい情報:

- `traceId`
- `userId`
- `articleId`
- `jobId`
- `eventName`
- `qualityStatus`
- `reasonCode`
- `TopicRisk`
- 文字数、見出し数、リンク数、引用数

ログへ出さない情報:

- 記事本文全文
- HTML本文全文
- プロンプト全文
- X投稿本文全文
- 外部APIレスポンス全文
- 秘密情報、認証情報、Webhook URL

監査ログには、公開投稿拒否、人間確認完了、公開操作、投稿ステータス変更などの操作結果を記録する。

## 19. テスト観点

| ID | 観点 | 期待結果 |
| --- | --- | --- |
| `QLT-001` | タイトル本文一致 | 本文と無関係なタイトルが`NeedsRevision`になる |
| `QLT-002` | 見出し重複 | 同一論点の重複見出しが検出される |
| `QLT-003` | 空本文 | 空本文が`ValidationError`になる |
| `QLT-004` | 禁止表現 | 根拠のない保証表現が警告される |
| `QLT-005` | 外部コピー | 長い引用や丸写しが差し戻される |
| `QLT-006` | strict再取得 | 最新性が必要な記事で再取得確認が要求される |
| `QLT-007` | compliance_strict | 人間確認前のPublishが拒否される |
| `QLT-008` | X引用再取得 | production/strictで再取得前のX引用が拒否される |
| `QLT-009` | X引用停止 | 削除、非公開、取得不能の投稿が引用停止になる |
| `QLT-010` | HTML安全性 | 危険HTMLを含む記事が投稿前に拒否または除去される |
| `QLT-011` | ログ除外 | 本文全文、プロンプト全文、X投稿本文全文がログに出ない |
| `QLT-012` | Draft既定 | WordPress投稿ステータス未指定時に`Draft`になる |
| `QLT-013` | 検索意図一致 | title、見出し、本文の主意図がずれる記事が差し戻される |
| `QLT-014` | 独自情報不足 | 検索結果要約だけの記事が`NeedsRevision`になる |
| `QLT-015` | E-E-A-T不足 | YMYL記事で著者、出典、更新日が不足すると`HumanReviewRequired`になる |
| `QLT-016` | AI量産リスク | 薄い派生ページ量産が警告される |
| `QLT-017` | 内部リンク品質 | 不明瞭なアンカーテキストが警告される |

テストでは実APIを呼ばない。Gemini、Tavily、X API、WordPressはモックまたはテストダブルで扱う。

## 20. 実装順序

1. 品質チェックの判定型をApplication層に定義する。
2. 記事構造、空本文、文字数のチェックを実装する。
3. タイトル、メタディスクリプション、見出し重複のチェックを実装する。
4. 検索意図、独自情報、E-E-A-Tのチェックを実装する。
5. 禁止表現とTopicRiskのチェックを実装する。
6. X引用再取得チェックを組み込む。
7. WordPress投稿前チェックへ品質判定を組み込む。
8. 画面に品質警告と人間確認状態を表示する。
9. `QLT-*`テストを追加する。

禁止表現やTopicRisk辞書は設定ファイルまたはDB管理を検討する。新しい本番依存関係を追加する場合は、実装タスクで確認する。

## 21. 決定事項

- AI生成結果は公開可能な完成物ではなく、品質チェック対象の下書きとして扱う。
- MVPでは品質結果の長期履歴を保存しない。
- X投稿は引用元カードとして表示できるが、事実確定の単独根拠にしない。
- SEOは人に役立つ独自情報を、検索エンジンとAI検索が理解しやすい形に整える活動として扱う。
- AI Overviews、AI Mode向けにも基本SEOを適用し、専用ハックを品質基準にしない。
- AI生成記事は独自情報、E-E-A-T、事実確認、人間編集を経て公開判断する。
- `compliance_strict`または`HumanReviewRequired`の記事は人間確認前にPublishしない。
- WordPress投稿の既定ステータスは`Draft`とする。
- 品質チェックはApplication層で行い、UI表示制御だけに依存しない。
